- **Key Idea:**

- **Caution:**
스택을 활용한 반복문 DFS 구현 시, 바깥쪽 루프의 인덱스( i )와 큐/스택에서 꺼낸 현재 노드의 인덱스( cur )를 혼동하여 참조하면 무한 루프나 잘못된 탐색이 발생할 수 있으므로 변수 스코프 관리에 극도로 주의해야 함.

- **Problem Hint:**
"A와 B가 연결되어 있고, B와 C가 연결되어 있으면 A와 C도 연결된 것.", " 네트워크 의 개수" -> 전형적인 '연결 요소( Connected Component )'의 개수를 구하는 그래프 탐색( DFS/BFS ) 문제임을 도출.

- **Portfolio Selling Point:**
대규모 데이터나 깊은 트리/그래프 구조를 다룰 때 발생할 수 있는 치명적인 콜 스택 오버플로우( Call Stack Overflow ) 크래시의 위험성을 인지하고, 이를 방지하기 위해 힙 메모리를 사용하는 std::stack 기반의 Iterative DFS를 직접 설계 및 디버깅해 낸 엔진 프로그래머로서의 메모리 안전성( Memory Safety ) 감각을 입증.

### PRG 43105 - 정수 삼각형
- **Topic:** DP | Bottom-Up | 1D Memoization
- **Folder:** `Week_03/04_DP/`

- **Space Complexity:** O( 1차원 DP 배열 메모리 )

- **Key Idea:**
위에서 아래로 내려가는 대신, 바닥에서부터 꼭대기로 올라가는 바텀업( Bottom-Up ) 방식을 채택.
현재 층의 노드가 가질 수 있는 최댓값은 바로 아래층의 연결된 두 자식 노드 중 더 큰 값을 현재 노드의 값에 더하는 방식으로 누적함.

- **Problem Hint:**
"삼각형의 꼭대기에서 바닥까지 이어지는 경로.", "거쳐간 숫자의 합이 가장 큰 경우" -> 경로의 부분합이 전체의 최적해를 구성하는 최적 부분 구조( Optimal Substructure )를 가짐을 파악하여 DP 알고리즘 도출.

- **Portfolio Selling Point:**
단순히 알고리즘을 푸는 것에 그치지 않고, 상태 전이의 규칙을 분석하여 2차원 배열을 1차원 배열로 압축하는 고도의 메모리 공간 최적화( Space Optimization ) 능력을 보여줌. 동시에 원본 인자( triangle )의 불변성( Immutability )을 유지하여 실무적인 API 설계 원칙을 준수하는 시각을 어필.

### PRG 42895 - N으로 표현
- **Topic:** DP | Set | Combinatorics
- **Folder:** `Week_03/04_DP/`
- **Time Complexity:** O(C) ( C는 N=8일 때 생성되는 모든 조합의 수, 상수에 가까움 )
- **Space Complexity:** O(K) ( K는 생성된 고유한 연산 결과값의 개수. unordered_set 메모리 )

- **Key Idea:**
숫자 N을 k번 사용해서 만들 수 있는 수의 집합을 DP[k]라고 할 때, DP[k]는 ( DP[i]의 원소 )와 ( DP[k-i]의 원소 )를 사칙연산한 결과들의 합집합임을 이용하여 상태 전이( State Transition )를 구현함. 추가로 N이 연속되는 수( N, NN, NNN... )를 미리 집합에 초기화함.

- **Caution:**
다중 루프가 5중으로 겹치기 때문에 인덱스 설계( 0-based 인덱스의 j와 i-j-1 분할 )가 하나라도 어긋나면 올바른 조합이 형성되지 않음. 또한, 메인 DP 연산뿐만 아니라 '연속된 숫지'를 배열에 초기화하는 단계에서도 목표 숫자( number )와 일치하는지 반드시 검사해야 함. 이를 누락하면 목표 숫자가 초기화 값( 예 : 55 )과 같을 때 정답을 무시하고 넘어가 오답( -1 )을 반환하는 논리적 오류가 발생함.

- **Improvement:**
가독성을 해치고 제어문 이탈 범위가 모호한 switch-case-break 구조를 if - else if - continue로 리팩토링하여 예기치 않은 쓰레기값 삽입 버그를 원천 차단함. 배열 초기화 구역에 단 한줄의 조기 탈출( Early Exit : if(nn = number ) return i + 1;) 코드를 추가하여 엣지 케이스 방어와 연산 속도 최적화를 동시에 달성함.

- **Problem Hint:**
"최솟값이 8보다 크면 -1을 return" -> 상태 공간( State Space )의 깊이가 8로 극단적으로 제한되어 있음을 파악하고, 깊이가 얕을 때는 다중 루프를 통한 모든 경우의 수 탐색( Exhaustive Search )과 DP의 결합이 최적의 해답임을 도출.

- **Portfolio Selling Point:**
단순히 알고리즘이 돌아가게 만드는 것을 넘어, 제어 흐름( Control Flow )을 실무 친화적으로 리팩토링하는 클린 코드 역량을 보여줌. 특히, 메인 루프에 진입하기 전 시스템의 '초기 상태( Initial State )'에 대한 검증 로직을 잊지 않고 구현하여, 놓치기 쉬운 엣지 케이스까지 완벽하게 잡아내는 꼼꼼한 QA( Quality Assurance ) 마인드를 입증함.

- **Improvement:**
전역 공간( Data Segment )에 할당된 2차원 벡터 배열( adj )은 로컬 환경에서 함수가 연속 호출될 때 이전 데이터가 누적되는 부작용을 일으킴. 코딩 테스트상에선 안전하지만 테스트가 불편하다면 adj[i].clear( )를 명시적으로 호출하여 시스템의 상태를 안전하게 초기화하는 방어 코드를 추가할 수 있음.

- **Problem Hint:**

짝수는 가장 마지막 비트가 무조건 0이므로 단순히 1을 더하는 것이 정답임. 홀수는 이진수 표기 시 뒤에서부터 연속된 1이 나오다가 처음으로 등장하는 0을 찾아 1로 바꾸고, 그 직전( 바로 오른쪽 ) 비트를 0으로 바꾸는 것이 최소 증가 폭임. 이를 수학적으로 최적화하면 `가장 낮은 0비트 = ~val & ( val + 1 )` 공식을 도출할 수 있으며, 기존 값에 이 비트를 더하고 절반으로 시프트 한 값을 빼면 짝/홀 분기 없이 단번의 수식으로 정답을 구할 수 있음.

- **Caution:**
비트를 직접 시프트 연산( << )하며 찾을 경우, 피연산자 val이 최대 10^(15)이므로 C++의 32비트 기본 리터럴 연산( 1 << idx )사용 시 오버플로우가 발생하여 쓰레기값이 도출됨. 반드시 64비트 리터럴인 1LL을 사용하여 오버플로우를 예방해야 함.

- **Portfolio Selling Point:**
일반적인 시뮬레이션 방식( while문을 이용한 비트 스캔 )의 한계와 32비트 메모리 오버플로우 이슈를 파악하고, 최적화의 극의인 분기 없는 O(1) 비트 조작식을 도출해 냄. 이는 게임 클라이언트 성능 최적화에서 CPU 연산 사이클을 극한으로 아끼는 역량을 입증하는 좋은 사례임.

- **Space Complexity:** O(N x M)

- **Key Idea:** 
오른쪽과 아래쪽으로만 이동할 수 있는 그리드 환경에서, 특정 칸 `(i, j)`에 도달할 수 있는 모든 최단 경로의 수는 "바로 위쪽 칸 `(i-1, j)`까지 오는 경우의 수" + "바로 왼쪽 칸 `(i, j-1)`까지 오는 경우의 수"의 합과 같다는 부분 문제의 중복성을 이용하는 전형적인 2차원 DP. 장애물이 존재하는 칸은 경우의 수를 강제로 0으로 초기화하여 상태 전이를 차단함.

- **Caution:**
문제 지문에서 물웅덩이 좌표가 `[m, n]` 즉, `[x( 가로 ), y( 세로 )]` 순서로 들어온다는 것을 간과하면 치명적인 런타임 에러나 오답이 발생함. 2차원 배열은 `arr[y][x]` ( 행, 열 ) 순서로 메모리에 접근하므로, 반드시 `dp[p[1]][p[0]]` 형태로 인덱스를 역전시켜 주어야 함. 또한, 값의 누적 속도가 매우 빠르므로 결과 반환 시가 아닌 루프 내부의 덧셈 연산 직후에 매번 `% MOD` 처리를 해주어야 오버플로우가 발생하지 않음.

- **Improvement:**
현재 패딩을 포함한 O(N x M) 크기의 2차원 배열을 사용 중이나, 점화식을 가만히 살펴보면 `i`번째 행을 계산할 때는 오직 `i-1`번째 행의 데이터만 필요함. 따라서 N 크기의 1차원 배열( Rolling Array 기법 ) 단 하나만 덮어씌워 가며 재사용한다면 공간 복잡도를 O(M) 수준으로 극단적으로 낮출 수 있음. ( 메모리가 극도로 제한된 임베디드나 쉐이더 환경에서 자주 쓰이는 최적화 기법 )

- **Problem Hint:**
1. 스케일 확인 : 가로 m, 가로 n은 최대 100. -> 전체 맵의 크기는 10,000. 탐색은 매우 여유로움.
2. 요구사항 분석 : 집에서 학교까지 가는 "최단 경로의 개수"를 구하라. -> 길을 '찾는' 것이라면 BFS가 적합하지만, 갈래길의 '조합의 수'를 구해야 하므로 이전 상태를 누적해 나가는 동적 계획법이 절대적으로 유리함.
3. 알고리즘 도출 :
    - 오른쪽, 아래쪽으로만 간다는 것은 "위, 왼쪽에서만 올 수 있다"는 뜻.
    - 점화식 : `DP[i][j] = DP[i-1][j] + DP[i][j-1]`
    - 예외 : 물웅덩이를 밟으면 안 되므로, 웅덩이 좌표의 DP 값은 0으로 만들어 값이 전파되지 않게 막으면 끝! 

- **Portfolio Selling Point:**
게임 엔진의 네비게이션 메시 설계 시, 특정 타일에 도달할 수 있는 다양한 경로의 가중치( Heatmap )를 계산하는 코어 로직을 입증함. 특히 `[x, y]` 순서로 주어지는 외부 데이터를 `[row, col]` 형태의 2차원 배열 인덱스로 정확히 파싱하여 공간 데이터 매핑 능력을 보여줌.

### PRG 49189 - 가장 먼 노드
- **Topic:** Algorithm | BFS | Graph
- **Folder:** `Week_07/01_Graph/`
- **Time Complexity:** O(V+E) ( 노드의 수 V와 간선의 수 E만큼만 정확히 탐색하므로 선형 시간에 수렴. )
- **Space Complexity:** O(V+E) ( 인접 리스트 구성 및 방문 배열 메모리 할당. )

- **Key Idea:**
간선의 가중치가 모두 동일한 그래프에서의 최단 경로는 다익스트라가 아닌 BFS로 접근하는 것이 가장 빠르다. 시작 노드에서 출발하여 큐에 이웃 노드를 삽입할 때마다 이전 노드의 거리 값에 +1을 누적( `visited[next] = visited[cur] + 1` )하면, 맵 전체로 물결이 퍼져나가듯 모든 노드에 대한 최단 거리를 O(V+E)만에 캐싱할 수 있다.

- **Caution:**
노드 번호가 1번부터 시작하므로 배열의 크기를 선언할 때 n이 아닌 n+1로 패딩을 주어야 인덱스 참조 에러가 발생하지 않음. 또한 양방향 그래프이므로 간선 정보를 삽입할 때 A->B뿐만 아니라 B->A도 반드시 추가해 주어야 함.

- **Improvement:**
현재 코드는 마지막에 `mxVal`과 같은 거리를 가진 노드를 카운트하기 위해 O(N) 순회를 한 번 더 수행하고 있음. BFS 특성상 가장 나중에 큐에서 나오는 노드들이 무조건 최장 거리이므로, 거리가 갱신될 때마다 카운트를 1로 초기화하고 동일한 거리일 때 카운트를 증가시키면 마지막 카운트 순회를 생략할 수 있음. 하지만 유지보수와 가독성 측면에서는 현재의 분리된 루프 구조가 매우 정석적이고 훌륭함.

- **Problem Hint:**
1. 스케일 확인 : 노드의 개수 n <= 20,000. 간선은 최대 50,000개. -> 2차원 배열( 인접 행렬 )로 그래프를 만들면 4억 칸의 메모리가 필요하므로 메모리 초기화 위험이 높음. 반드시 연결된 정보만 저장하는 '인접 리스트' 구조를 써야 함.
2. 요구사항 분석 : 1번 노드에서 가장 멀리 떨어진 노드의 갯수를 구하라. -> 모든 간선의 길이가 1로 동일하므로, 큐를 이용한 BFS 탐색으로 거리를 갱신해 나가는 것이 최적임.
3. 알고리즘 도출 :
    - 1번 노드부터 큐에 넣고 BFS를 돌면서, 아직 방문하지 않은 이웃 노드를 만날 때마다 `visited[이웃] = visited[현재]  +1`을 기록함.
    - 큐에 삽입할 때마다 `mxVal`을 갱신하면 자연스럽게 최대 거리가 도출됨..

- **Portfolio Selling Point:**
가중치가 동일한 그래프 환경에서 최적의 탐색 알고리즘( BFS )을 선택하고, 메모리 단편화를 막기 위해 인접 행렬 대신 인접 리스트를 직접 구현하여 데이터 리소스 관리 능력을 입증함.

### PRG 118669 - 등산코스 정하기
- **Topic:** Algorithm | Graph | Dijkstra
- **Folder:** `Week_07/01_Graph/`
- **Time Complexity:** O(E log V) ( 단 한 번의 다익스트라 실행으로 모든 간선( E )을 우선순위 큐( log V )에 넣고 빼며 탐색함.)
- **Space Complexity:** O(V+E) ( 인접 리스트 배열 및 최댓값을 ㅗ초기화된 상태 배열 d 메모리 할당. )

- **Key Idea:**
기존의 다익스트라가 '거리의 누적 합'을 최소화하는 것이라면, 이 문제는 경로 상의 '가장 큰 가중치( 병목 )'를 최소화하는 Min-Max Path 문제임. 출입구들을 모두 큐에 넣고( 다중 출발점 ), 상태 전이 점화식을 `d[next] = max( 현재까지의 최대 난이도, 다음 간선 가중치 )` 로 변경함. 또한 큐에서 꺼낸 노드가 '산봉우리'일 경우 이웃 탐색을 중단하여 편도 최적 경로만 구하도록 가지치기를 수행함.

- **Caution:**
C++ STL `priority_queue` 사용 시, `const auto& cur = pq.top(); pq.pop();` 형태로 작성하면 `cur`가 삭제된 메모리를 가리키는 `Dangling Reference`가 되어 Undefined Behavior가 발생함. 요소를 빼내어 계속 사용해야 할 때는 참조자를 빼고 `const auto cur`로 값 복사를 수행해야 함.

- **Improvement:** 결과를 도출할 때 `1`부터 `n`까지 전체 노드를 순회하며 `is_summit`을 확인하는 것도 훌륭하지만, 주어지는 `summits` 배열 자체를 오름차순으로 정렬한 뒤 `summits` 배열만 순회하면 탐색 횟수를 O( 봉우리 개수 )로 미세하게 줄일 수 있음. 단, 현재 코드의 가독성과 로직 흐름이 매울 정석적이므로 수정할 필요는 없음.

- **Problem Hint:**
1. 스케일 확인 : 노드( N ) 5만, 간선 20만 개. -> 모든 출발지에서 모든 목적지까지 완전 탐색을 수행하면 무조건 시간 초과 발생. 단 한 번의 다익스트라 탐색으로 끝내야 함.
2. 요구사항 분석 : 출발지에서 목적지까지 가는 경로 중 '가장 큰 가중치'의 최솟값을 구하라. -> 거리를 누적하는 대신, 지나온 간선 중 최댓값을 기억하며 진행.
3. 알고리즘 도출 :
    - 출발지가 여러 개이므로, 시작 전 모든 출발지의 `d` 값을 0으로 세팅하고 큐에 삽입.
    - `new_intensity = max(cur.intensity, next.weight)`로 계산.
    - `if (d[next] <= new_intensity)` 일 경우 스킵, 엄격하게 작아질 때만 갱신.
    - 도착지에 도달하면 더 이상 큐에 이웃을 넣지 않고 종료.

- **Portfolio Selling Point:**
네트워크 패킷 분석이나 엔진 렌더링 파이프라인에서 발생하는 병목 구간 최소화 경로를 탐색하는 Min-Max Dijkstra 아키텍처를 완벽히 구현함. 레퍼런스 수명 관리에 대한 C++ 메모리 이해도를 증명함.

### PRG 12971 - 스티커 모으기 2
- **Topic:** Algorithm | Dynamic Programming
- **Folder:** `Week_07/02_DP/`
- **Time Complexity:** O(N) ( 길이가 N인 배열을 정확히 2번 순회하므로 선형 시간에 수렴함. )
- **Space Complexity:** O(N) ( N 크기의 `dp1`, `dp2` 벡터 2개를 할당함. )

- **Key Idea:**
원형으로 연결된 구조에서 인접한 요소를 선택할 수 없는 제약을 풀기 위한 전형적인 테크닉. 배열이 원형일 때는 '첫 번째 요소를 선택하면 마지막 요소를 절대 선택할 수 없다'는 절대 규칙이 생김. 따라서 1) 첫 번째 슽티커를 뜯는 경우( `dp1` : 마지막 원소 전까지만 탐색 ), 2) 첫 번째 스티커를 뜯지 않는 경우( `dp2` : 마지막 원소까지 탐색 )로 평행 우주를 나누어 1차원 배열처럼 점화식 DP[i] = max(DP[i-1], DP[i-2] + sticker[i])를 전개한 뒤, 두 결과 중 최댓값을 취함.

- **Caution:**
입력 데이터 크기 N이 1인 예외 케이스를 반드시 처리해야 함. N = 1일 경우,, `dp1[1] = dp1[0]` 같은 초기화 로직에서 배열 범위를 벗어나는 런타임 에러가 발생할 수 있으므로, 루프 진입 전 `if (n == 1)`을 통해 조기 반환해야 함.

- **Improvement:**
현재는 N 크기의 `vector<int>` 2개를 사용하여 O(N)의 공간 복잡도를 가지지만, 점화식을 살펴보면 현재 값을 구하기 위해 오직 직전 값과 두 칸 전 값 딱 2개의 변수만 필요함. 따라서 배열 전체를 메모리에 올릴 필요 없이, `prev1`, `prev2` 와 같은 스칼라 변수 2개만 선언하여 롤링시키면 공간 복잡도를 O(1) 수준으로 극단적으로 낮출 수 있음.

- **Problem Hint:**
1. 스케일 확인 : 스티커의 길이 N <= 100,00. ->  2중 루프(O(N^2)) 절대 불가. 한 번의 순회로 끝내는 O(N) 알고리즘( DP 또는 그리디 )이 강제됨.
2. 요구사항 분석 : 스티커를 뜯어내어 얻을 수 있는 숫자의 합의 최댓값을 구하라. 단, 인접한 스티커는 뜯을 수 없고 배열은 원형으로 연결되어 있다.
3. 알고리즘 도출 :
    - '인접한 것을 고를 수 없다'는 제약은 전형적인 계단 오르기/도둑질 유형의 DP임.
    - 원형 제약을 끊어내기 위해 첫 번째 원소 선택 여부에 따라 배열을 2개로 나누어 전개함.

- **Portfolio Selling Point:**
원형 휠 메뉴나 클로즈드 루프 트랙 같은 게임 내 원형 자료구조에서 발생하는 상태 의존성 문제를 2개의 상태 공간으로 분할하여 O(N)으로 최적화하는 수학적 모델링 능력을 입증함.

### PRG 01832 - 보행자 천국
- **Topic:** Algorithm | Dynamic Programming | 3D State
- **Folder:** `Week_07/02_DP/`
- **Time Complexity:** O(M*N) ( M * N 크기의 그리드를 정확히 한 번 순회하므로 선형 시간에 수렴함. )
- **Space Complexity:** O(M*N) ( `dp[m][n][2]` 형태의 3차원 벡터를 할당하여 상태를 캐싱함. )

- **Key Idea:**
목적지에 도달하는 경우의 수를 구하는 전형적인 2D 길찾기 DP 문제이지만, `2`라는 '관성 유지( 직진 통행만 가능 )' 조건 때문에 "어느 방향에서 이 칸을 진입했는가?"라는 과거의 상태가 현재의 이동 가능 여부를 결정함. 이를 해결하기 위해 DP 테이블을 3차원으로 찢어 `dp[i][j][0]`( 가로 진입 )과 `dp[i][j]1]`( 세로 진입 )로 분할함.
점화식을 세울 때는 '현재 내 칸'의 상태가 아니라 '나에게 오려는 이전 칸'의 상태를 물어보는 방식으로 전개함. ( 예 : 왼쪽 칸이 `2`라면 왼쪽 칸에 가로로 진입했던 경우의 수만 받아올 수 있음. )

- **Caution:**
첫 번째 행과 첫 번째 열을 초기화할 때, 중간에 벽( `city_amp == 1` )을 만나면 그 이후의 칸들은 절대 도달할 수 없으므로 반드시 `break`를 걸어주어야 함. 또한 값을 누적하는 덧셈 구간뿐만 아니라 값을 그대로 물려받는 구간( `else if == 2` )에서도 오버플로우 방지를 위해 반드시 `%MOD`를 붙여주어야 안전함.

- **Improvement:**
현재 `vector<vector<vector<int>>>`를 사용하여 3차원 배열을 할당하고 있음. 실무에서 맵의 크기가 10,000 이상으로 커질 경우 메모리 파편화가 발생할 수 있으므로, `vector<vector<int>> right_dp, down_dp` 2개의 2차원 배열로 평탄화하거나, 3차원 `vector` 대신 1차원 `vector`에 인덱스 수학 연산( `y*N*2 + x*2 + dir` )을 적용하면 캐시 히트율을 극대화할 수 있음. 하지만 현재 문제 스케일에서는 가독성이 가장 뛰어난 완벽한 코드임..

- **Problem Hint:**
1. 스케일 확인 : m, n <= 500. -> O(N^2) 완전 탐색( BFS/DFS ) 시 경우의 수가 기하급수적으로 폭발. 부분 구조 최적화가 가능한 DP를 사용해야 함.
2. 요구사항 분석 : 출발지에서 목적지까지 도달하는 모든 경로의 수를 구하라. 단, `2`칸에서는 직진만 가능하다.
3. 알고리즘 도출 :
    - '방향성' 자체가 중요한 상태가 되므로, 2차원 좌표에 방향 차원을 추가한 3D DP를 설계함.
    - 직전 칸이 0( 자유 ), 1( 벽 ), 2( 직진 )일 때의 3가지 분기를 가로/세로 각각 나누어 O(1)로 계산함.

- **Portfolio Selling Point:**
네비게이션 메시에서 유닛의 회전 반경이나 관성 물리를 고려한 Path-Finding 로직 설계 시, 이전 프레임의 상태 벡터를 현재 연산에 결합하여 문제를 해결하는 아키텍처 역량을 입증함.

### PRG 43238 - 입국심사
- **Topic:** Algorithm | Parametric Search
- **Folder:** `Week_07/03_ParametricSearch/`
- **Time Complexity:** O(M log(MaxTime * N)) ( M은 심사관의 수. 최대 10^18까지의 시간 범위를 이분 탐색하므로 60회 내외의 루프만 돌며, 각 루프에서 M만큼 순회함. )
- **Space Complexity:** O(1) ( 추가적인 배열 할당 없이 스칼라 변수만 사용함. )

- **Key Idea:**
전형적인 Parametric Search( 매개 변수 탐색 ) 문제. "모든 사람을 심사하는 데 걸리는 최소 시간"이라는 최적화 문제를 "주어진 시간 `mid`안에 모든 사람을 심사할 수 있는가?"라는 결정 문제로 변환함.

- **Caution:**
C++ 자료형의 한계를 명확히 인지해야 함. 탐색의 ed 값은 최대 10억( 시간 ) x 10억( 명 ) = 10^18까지 커질 수 있으므로, st, ed, mid 및 누적 인원수 cnt는 반드시 long long 타입으로 선언해야 함. 이를 32비트 int로 선언할 경우 엄청난 오버플로우가 발생하여 무한 루프에 빠지거나 오답을 냄.

- **Improvement:**
현재 `solve` 함수 내부의 `for`문에서 모든 심사관을 순회하며 `cnt`를 누적하고 있음. 만약 심사관이 10만 명이고 앞부분의 몇 명만 계산해도 이미 `cnt`가 `n`을 초과했다면, 뒤의 나머지 심사관을 계산할 필요가 없음.
따라서 `for`문 내부에 `if(cnt >= n) return true;` 라는 단 한 줄의 조기 반환 코드를 추가하면, 최악의 경우에도 연산량을 극단적으로 줄이는 마이크로 최적화가 가능함.

- **Problem Hint:**
1. 스케일 확인 : 사람 수 N <= 10^9, 심사관 처리 시간 <= 10^9. -> 절대로 선형 시간( O(N) )으로 사람 한 명씩 시뮬레이션 할 수 없음.. 무조건 이분 탐색( O(log N) )으로 범위를 좁혀야 함.
2. 요구사항 분석 : "모든 사람을 심사하는 최소 시간"을 구하라. -> "X 시간이 주어지면 N 명을 처리할 수 있는가?( Yes/No )"로 질문을 비틀어 Parametric Search를 설계함.
3. 알고리즘 도출 :
    - st는 0, ed는 최악의 경우( 가장 느린 심사관 * N명 )로 설정..
    - mid = ( st + ed ) / 2 일 때, 모든 심사관이 mid 시간 동안 심사할 수 있는 사람 수의 합을 구함( mid / times[i] ).
    - 합산 인원이 N명 이상이면 ed = mid, 미만이면 st = mid + 1로 갱신함.

- **Portfolio Selling Point:**
네트워크 패킷이나 서버 스레드 풀이 대규모 트래픽을 분산 처리할 때, "할당된 시간 타임아웃 내에 처리 가능한가?"를 묻는 이분 탐색 로직으로 역산하여 스케줄링 문제를 O(N)에서 O(log N)으로 획기적으로 최적화하는 아키텍처 능력을 증명함.

### PRG 64062 - 징검다리 건너기
- **Topic:** Algorithm | Parametric Search | Binary Search
- **Folder:** `Week_07/03_ParametricSearch/`
- **Time Complexity:** O(N log M) ( `N`은 배열의 길이 최대 20만, `M`은 돌의 내구도 최대 2억. 약 28번의 로그 스케일 탐색마다 선형 탐색을 수행함. )
- **Space Complexity:** O(1) ( 별도의 컨테이너 복사 없이 인덱스 스칼라 변수만 사용. )

- **Key Idea:**
`최적화 문제`를 `결정 문제`로 뒤집는 것이 핵심. `X명`이 건너기 위해 밟아야 할 돌의 최소 내구도는 `X`이므로, `stones[i] >= x`인 돌들만 밟고 지나간다고 가정함. 이때 `밟은 돌과 직전 돌의 간격 ( i - idx )`이 `k`를 초과하는 구간이 단 하나라도 존재하면 무사히 건너는 것에 실패( `false` )한 것으로 판정함.

- **Caution:**
`solve` 함수의 루프가 끝난 뒤, 마지막으로 밟은 돌에서 `도착지( 반대편 기슭 )` 까지의 점프 거리도 반드시 검사해야 함. 도착지의 인덱스는 `stones.size()`에 해당하므로, 마지막 반환문에 `(int)stones.size() - idx <= k`를 누락하면 엣지 케이스에서 오답이 발생함.

- **Improvement:**
Parametric Search 관점에서는 완벽한 코드임. 만약 연산 속도를 극한으로 끌어올리고 싶다면,, 이분 탐색을 걷어내고 `덱( Deque )`을 활용한 `Sliding Window Maximum` 알고리즘을 도입하여 `O(N)` 선형 탐색 단 한 번으로 문제를 뚫어내는 방식으 시도해 볼 수 있음.

- **Problem Hint:**
1. 스케일 확인 : 돌 개수 `N = 200,000`, 내구도 `M = 200,000,000`. 사람을 1명씩 건너게 하는 선형 시뮬레이션 `O(N * M)` 시 시간 초과 확정. 무조건 `이분 탐색`이 강제됨.
2. 요구사항 분석 : "최대 몇 명이 건널 수 있는가?" -> "`x명`이 주어졌을 때, 못 건너는 돌이 연속으로 `k`개 이상 나오는가?( Yes/No )"
3. 알고리즘 도출 : `st = 0, ed = 최대 내구도`로 두고 탐색 시작. 사람 수가 `mid`일 때 조건이 성립하면 더 많은 인원을 시도하기 위해 `st = mid`로 갱신, 불가능하면 인원을 감축하기 위해 `ed = mid - 1`로 갱신함.

- **Portfolio Selling Point:**
해결이 불가능해 보이는 복잡도 `O(N * M)`의 로직을, 결정 문제로의 치환을 통해 `O(N log M)`으로 대폭 축소하는 아키텍처 설계 능력을 입증. 또한, 상태 추적 시 불필요한 배열이나 카운터를 쓰지 않고 `Index Tracking`만으로 메모리 및 연산 부하를 최소화하는 엔진 지향적 코딩 스타일을 보여줌.

### PRG 17685 - 자동완성
- **Topic:** Algorithm | String | Trie
- **Folder:** `Week_07/04_StringSearch/`
- **Time Complexity:** O(N * L) ( `N`은 단어의 개수, `L`은 단어의 최대 길이. 구축과 탐색 모두 단어 길이만큼만 순회하므로 선형 시간에 수렴함. )
- **Space Complexity:** O(N * L * 26) ( 최악의 경우 모든 단어가 한 글자도 겹치지 않을 때 생성되는 노드의 최대 개수. )
- **Key Idea:**
수많은 문자열 중 특정 단어를 찾기 위해, 문자열의 `접두사`를 트리의 간선으로 공유하는 `Trie( 트라이 )` 자료구조를 구축함. 단어들을 삽입할 때 해당 노드를 지나가는 횟수( `cnt` )를 누적해 둠. 이후 탐색 과정에서 트리를 타고 내려가다가, `cnt == 1`인 노드를 만나는 순간 "이 경로를 타는 단어는 나 하나뿐이다"라는 의미이므로 즉시 탐색을 종료하고 타이핑 횟수를 정답에 합산함.

- **Caution:**
구조체 내부의 포인터 배열 `Node* child[26]`은 명시적으로 초기화하지 않으면 쓰레기값이 들어갈 위험이 있음. C++에서 `new Node()`로 호출하여 Zero-Initialization이 운 좋게 적용되었으나, 실무에서는 구조체 내부에 `Node* child[26] = {nullptr};` 처럼 명시적으로 널 포인터 초기화를 걸어두는 것이 발생 가능한 치명적 메모리 참조 에러를 막는 가장 안전한 방법임.

- **Improvement:**
엔진의 파이프라인 관점에서 `new Node()`를 통한 동적 할당은 시스템 콜을 유발하여 병목의 원인이 될 수 있음. 속도를 극한으로 끌어올리려면 배열 기반의 `정적 트라이`를 선언하여 풀링 방식으로 노드를 당겨쓰는 기법( `int child[MAX_NODES][26]` )을 적용해 볼 수 있음. 하지만 객체지향적 가독성과 유지보수 측면에서는 현재의 포인터 기반 트라이가 표준적인 정석 코드임.

- **Problem Hint:**
1. 스케일 확인 : 단어 개수 최대 `100,000`개. 총 문자 수 `1,000,000`개. 단순 `for`문으로 `substr()` 비교를 남발하면 무조건 시간 초과 발생.
2. 요구사항 분석 : 단어를 타이핑하다가 남은 단어가 하나뿐일 때( 유일한 접두사 ) 타이핑을 멈춤. 총 타이핑 횟수의 합을 구하라.
3. 알고리즘 도출 : 단어의 `접두사` 빈도를 트리 형태로 추적해야 하므로 `Trie` 자료구조가 완벽히 들어맞음. 노드에 방문 횟수를 저장하여 `1`이 되는 순간 탐색을 끊어내는 로직을 설계.

- **Portfolio Selling Point:**
콘솔 자동완성, 대규모 에셋 이름 검색 등 문자열 매칭 병목이 발생하는 시스템에서 `Trie 자료구조`를 직접 아키텍처링하여 검색 복잡도를 `O(N)`에서 `O(L)` ( 단어 길이 ) 수준으로 압도적으로 단축시키는 메모리-시간 교환 최적화 능력을 증명함.

### PRG 67258 - 보석 쇼핑
- **Topic:** Algorithm | Two Pointers | Hash Map
- **Folder:** `Week_08/01_Two_Pointers/`
- **Time Complexity:** O(N) ( `N`은 보석 배열의 길이 최대 10만. `left`와 `right` 포인터가 각각 배열을 최대 한 번씩만 순회하므로 선형 시간에 수렴함. Hash Map 삽입/삭제는 평균 `O(1)`.)
- **Space Complexity:** O(K) ( `K`는 고유한 보석의 종류 수. `set`과 `unordered_map`을 유지하기 위한 공간.)

- **Key Idea:**
`모든 종류의 보석을 포함하는 가장 짧은 구간`을 찾기 위해 두 개의 포인터( `left`, `right` )를 조작함.
루프 내부의 로직을 3가지 상태로 분리하여 제어하는 것이 핵심.
1. `모든 보석을 모은 상태 ( cnt == mx_cnt )` : `left` 포인터를 우측으로 당기며 윈도우를 수축시킴. 이때 뺄 수 있는 보석( `count > 1` )은 미련 없이 빼고, 빼면 조건이 깨지는 보석( `count == 1` )을 만났을 때 비로소 현재 구간의 길이를 정답 후보와 비교 및 갱신함.
2. `보석이 부족한 상태 ( cnt != mx_cnt && right < size )` : `right` 포인터를 우측으로 밀며 새로운 보석을 맵에 추가( 확장 )함.
3. `종료 상태` : 보석이 부족한데 `right`가 끝에 도달했다면, `left`를 당겨봤자 보석은 더 줄어들 뿐이므로 루프를 탈출( `break` )함.

- **Caution:**
`right` 포인터는 항상 `다음에 추가할 보석의 인덱스`를 가리키고 있으므로, 현재 윈도우에 포함된 실제 보석의 개수( 구간 길이 )는 `right - left`임. 길이를 비교할 때 기준점의 Off-by-one(+-1) 에러가 나지 않도록 `( right - 1 ) - left`등 본인만의 일관된 길이 비교 공식을 유지해야 함.

- **Improvement:**
현재 코드는 아키텍처 관점에서 완벽한 최적화 상태임. 실무의 극단적인 환경에서 `O(N)` 내부의 Hasm Map( `unordered_map` ) 접근으로 인한 오버헤드조차 줄이고 싶다면, 초반에 `set`으로 구한 고유 보석들을 `vector<string>` 등에 넣어 `정수 ID( 0 ~ K-1 )`로 맵핑한 뒤, `unordered_map` 대신 `int count[100000] = {0, }` 형태의 단순 `정수 배열`을 해시 테이블처럼 사용하면 연산 속도를 3배 이상 폭발적으로 끌어올릴 수 있음.

- **Problem Hint:**
1. 스케일 확인 : 배열 길이 `N = 100,000`. 이중 `for`문으로 모든 구간을 탐색하면 `O(N^2) = 100억`번의 연산으로 무조건 시간 초과 발생. 선형 탐색( `O(N)` )이 강제됨.
2. 요구사항 분석 : "모든 종류를 포함하는 가장 짧은 구간" -> 구간의 확장은 `right`가, 구간의 수축은 `left`가 담당하는 `투 포인터` 로직이 완벽하게 부합함.
3. 알고리즘 도출 : `while(true)` 형태의 무한 루프 내부에 수축/확장/탈출 분기를 명확하게 설계하고, `unordered_map`을 통해 윈도우 내의 보석 개수를 `O(1)`로 트래킹함.

- **Portfolio Selling Point:**
복잡한 상태 전이가 일어나는 알고리즘에서, 각각의 조건이 충돌하지 않도록 명확한 3단 상태 머신 분기로 코드를 제어하는 역량을 보여줌. 특히, 데드락이나 무한 루프에 빠지기 쉬운 포인터 동기화 로직을 우아하게 해결함.

- **Time Complexity:** O(2^C * R * C) ( `C`는 컬럼 수 최대 8, `R`은 로우 수 최대 20. 비트 부분집합 탐색 `2^8 = 256`회 루프 내부에서 각 로우의 문자열을 이어 붙이는 연산이 일어남. 최대 연산량이 약 4만 회에 불과하여 매우 빠름. )
- **Space Complexity:** O(R * L) ( `L`은 문자열 길이. 유일성 검사를 위해 각 루프마다 `set`에 문자열을 임시 저장하는 공간. )

- **Key Idea:**
속성의 개수가 매우 작다( 최대 8개 )는 점에 착안하여, 속성의 선택 상태를 비트마스킹으로 0부터 255까지의 정수로 치환함 ( `00000001(2) ~ 11111111(2)` ).
모든 조합에 대해 순회하며 두 가지 조건을 검증함.
1. `유일성` : 현재 켜진 비트에 해당하는 속성들의 문자열을 이어 붙여( Concatenation ) `set`에 삽입하고, 최종 `set`의 크기가 원본 테이블의 행 개수와 일치하면 식별 가능한 키로 인정.
2. `최소성` : 유일성을 통과한 키가, 이미 정답으로 등록된 `기존의 키`를 포함하고 있는지( 확장판인지 ) 검사함. `( 기존키 & 새로운 키 ) == 기존키` 공식을 활용하여 단 한 번의 비트 AND 연산으로 부분집합 포함 여부를 검증함.

- **Caution:**
비트가 켜져 있는지 확인하는 마스킹 조건문 작성을 가장 주의해야 함. `if ( (i & (1 << k)) == 1 )`이라고 쓰면 치명적인 논리 오류가 발생함. `i = 0010(2)` 이고 `k = 1`일 때, `i & (1 << 1)`의 결과는 불리언 True( 1 )가 아니라 정수 `2`가 튀어나오기 때문임. 따라서 값을 직접 비교하지 말고 `if ( i & (1 << k) )` 처럼 암시적 0( False ) / 비 0( True ) 평가를 활용하거나, 굳이 명시하려면 `if ( (i & (1 << k)) != 0 )`으로 작성해야 함.

- **Improvement:**
ㅎ녀재 코드는 비트마스킹의 정석을 그대로 밟고 있는 훌륭한 형태임. 속도를 더 올리기 위해 굳이 문자열 덧셈( `str += relation ...` )을 하지 않고 튜플 자체를 해싱하는 기법도 있으나, 이 문제에서는 R과 C의 스케일이 워낙 작기 때문에 현재의 직관적인 `set<string>` 방식이 유지보수 관점에서 훨씬 유리함. `keys.empty()` 분기 처리는 사실 불필요함( 루프 자체가 안 돌기 때문 ). 바로 `for`문으로 최소성 검사를 돌려도 무방함.

- **Problem Hint:**
1. 스케일 확인 : 컬럼 수 `C \le 8`. 이 숫자를 보는 순간 "비트마스킹 상태 압축을 써라"는 출제자의 강렬한 의도를 눈치채야 함.
2. 요구사항 분석 : "튜플을 식별할 수 있는가( 유일성 )?", "꼭 필요한 속성만 남겼는가( 최소성 )?" 두 개의 파이프라인을 설계해야 함.
3. 알고리즘 도출 : `1`부터 `(1 << C) - 1`까지 탐색하며 `set`으로 유일성을 먼저 뚫고, 통과한 녀석들만 정답 배열에 모아둔 기존 키들과 비트 AND 연산( `&` )을 돌려 최소성을 검증함.

- **Portfolio Selling Point:** 
다중 속성의 상태 조합을 배열이 아닌 단일 비트마스크로 압축하여 메모리 풋프린트를 최소화하고, 집합의 포함 여부를 O(N) 반복문이 아닌 단일 CPU 비트 연산 사이클로 끝내버리는 극한의 Low-level 최적화 역량을 입증함.

1. 스케일 확인 : 자물쇠 및 열쇠 크기 `N, M <= 20`. 이 정도로 극단적으로 작은 숫자는 100% 확률로 `무식하게 다 해보는 완전 탐색( Brute-Force )`이 정해임.
2. 요구사항 분석 : "열쇠를 돌리고 밀어서 자물쇠와 완벽히 맞물리게 하라" -> 2D 배열의 `90도 회전 공식` 및 `이동 오프셋` 처리가 핵심 요구사항.

- **Portfolio Selling Point:**
단순히 `if-else` 분기문으로 경계선 예외 처리를 남발하는 조잡한 코딩 대신, `공간 확장` 아키텍처와 `데이트 롤백` 패턴을 도입하여 복잡한 물리 충돌 시스템을 단 한 장의 파이프라인으로 안전하게 구현해 내는 대규모 시뮬레이션 설계 능력을 증명함.

초기 그래프 빌드 단계에서 `cables`에 없는 역방향 경로라도 반드시 `adj[v].push_back(u)`를 수행해야 함. 이를 누락하면 나중에 `flow[v][u]`의 잔여 용량이 양수로 변하더라도, BFS가 탐색할 수 있는 통로 배열에 존재하지 않아, 환불 로직이 작동하지 않고 조기에 멈추는 대형 버그가 발생함.

- **Improvement:**
현재 V = 500 스케일에서는 500 x 500 사이즈의 2D `vector` 테이블이 메모리상에서 연속적으로 배치되므로 캐시 히트율이 좋아 성능이 잘 나옴.

- **Problem Hint:**
1. 스케일 확인 : 노드 수 `N <= 500`, 간선 수 `M <= 10,000`. 데이터의 흐름과 한계치( `Capacity` )가 주어짐.
2. 요구사항 분석 : 단일 경로 매칭이 아니라, 네트워크 전체의 `동시 전송량 최댓값`을 요구함 -> 100% 네트워크 유량 문제.
3. 알고리즘 도출 : 에드몬드-카프 알고리즘을 선택하고, 정/역방향 간선을 인접 리스트에 동시 장착한 뒤 `while(BFS)` 파이프라인 가동.

- **Portfolio Selling Point:**
단순히 최단 거리를 찾는 초급 수준의 그래프 탐색을 넘어, 자원의 분산 제어와 최적 배분( Resource Allocation )의 핵심인 `잔여 그래프( Residual Graph )` 및 `음의 유량 회계 처리` 기법을 완벽하게 이해하고 하드웨어 레벨로 구현해 내는 하이엔드 최적화 역량을 증명함.

### PRG 92343 - 양과 늑대
- **Topic:** Algorithm | DFS | Graph | Bitmask
- **Folder:** `Week_09/04_StateSpaceSearch/`

- **Key Idea:**
기존의 DFS/BFS처럼 트리의 간선을 따라 위아래로 물리적인 이동( Walk )을 구현하는 것이 아님.
대신 내가 현재 서 있는 위치를 기반으로 텔레포트할 수 있는 후보 노드들의 집합( State )을 관리하는 것이 핵심.

2. 내 자식 노드들을 새롭게 갈 수 있는 후보 리스트에 편입 : next_state |= ( 1 << child );
3. 갱신된 후보 리스트( next_state )에 켜진 비트들을 순회하며, 조건( sheep > wolf )이 허락하는한 자유롭게 텔레포트 DFS를 전개.

- **Caution:**
비트 연산자( |= )와 비교 연산자( != )의 형태가 비슷하여 오타가 발생할 경우, 자식 노드의 비트가 갱신되지 않고 허공에 날아가 탐색이 조기 종료되는 치명적인 논리 에러가 발생함. C++에서 비트를 조작할 때는 반드시 대입( = )이 포함되어 있는지 확인해야 함.
또한, 테스트 케이스를 순차적으로 실행하는 main 함수 환경에서는 전역 변수( max_sheep, adj )의 상태 오염이 발생하기 쉬우므로 반드시 solution 시작 지점에서 강제 초기화를 수행해야 함.

- **Improvement:**
현재의 로직은 정답 도출에 완벽하며, 추가적인 자료구조나 함수 포인터 최적화 없이도 실무에서 가장 선호되는 클린 코드 형태를 띠고 있음. 만약 N >= 30 이상으로 스케일업된다면, 방문했던 비트마스크 상태 자체를 기록하여 중복 탐색을 막는 메모이제이션( DP with Bitmasking )을 결합해야 함.

- **Problem Hint:**
1. 스케일 확인 : N <= 17. 매우 작다. -> 시간 복잡도 O(2^N) 또는 O(N * 2^N) 대역의 완전 탐색/비트마스킹이 해답임을 유추.
2. 요구사항 분석 : 한 번 획득한 동물은 사라지며, 부무 노드를 거쳐 다시 내려가는 '재방문'이 허용됨.
3. 알고리즘 도출 : 물리적 이동 동선을 버리고, "방문 가능한 다음 노드 집합"을 들고 다니는 DFS 구조 채택. 이를 비트마스크로 제어하여 성능 극대화.

- **Portfolio Selling Point:**
일반적인 트리 탐색 문제의 함정( 물리적 재방문 구현 )에 빠지지 않고, 문제의 본질을 '상태 공간의 전이'로 재해석하는 아키텍처 설계 능력을 증명함. 나아가 배열의 동적 할당 오버헤드를 막기 위해 비트마스킹을 능수능란하게 활용하여 최상위 수준의 메모리 관리 및 CPU 연산 최적화를 이뤄냄.

- **Time Complexity:** O(N^2) ( `s`의 길이 `n <= 1000`. 외부 루프가 최대 `N/2`번 돌고, 내부 루프가 최대 `N/unit`번 돌며 `substr` 복사 오버헤드가 `O(N)` 발생하므로, 최악의 경우에도 연산량이 약 `500 * 1000 = 500,000`번에 수렴하여 0.0005초 이내에 통과함. )
- **Space Complexity:** O(N) ( 각 단위별로 압축된 임시 문자열을 생성하여 메모리에 저장하는 `answerStr`용 공간 복잡도. )

- **Key Idea:**

- **Caution:**

- **Problem Hint:**
1. 스케일 확인 : 데이터 크기가 `n <= 1000`으로 매우 작음 -> 임시 문자열 객체를 무수히 생성하는 `O(N^2)` 완전 탐색 알고리즘을 설계해도 무방함을 인지.
2. 요구사항 분석 : 문자열은 항상 `맨 앞부터 고정된 단위로만` 잘라야 함. 중간에서부터 임의로 자르는 예외는 없음.

- **Caution:**
구조체 내부에 배열을 선언할 때, `TrieNode* child[26]'`처럼 초기화 없이 방치하면 쓰레기 값이 들어가 `if (child == nullptr)` 검증이 무력화되고 즉각적인 세그먼테이션 폴트가 발생한다. 반드시 `= { nullptr }`를 명시해야 한다.

- **Improvement:**
코딩 테스트 환경에서는 메모리가 넉넉하여 매번 `new` 연산자로 힙 메모리를 동적 할당해도 무방하나, 실제 게임 엔진이나 대규모 트래픽 서버에서는 잦은 `new/delete`가 메모리 다편화를 유발한다. 따라서 세그먼트 트리에서 사용했던 것처럼 메모리 풀 방식인 `vector<TrieNode> node_pool; node_pool_.reserve(MAX_NODES);`를 활용해 연속된 공간에 사전 할당해 두고 참조하는 방식으로 업그레이드할 수 있다.

- **Problem Hint:**
1. 스케일 확인 : N = 10,000, M = 100,000. O(N * M) 문자열 비교는 10억 번 연산으로 무조건 시간 초과.
2. 요구사항 분석 : 주어진 문자열이 사전에 등록된 단어들의 '접두사'인지 빠르게 확인하고 그 개수를 집계해야 함.
3. 알고리즘 도출 : 문자열 검색에 특화된 극상위 알고리즘 `Trie( 트라이 )` 구조체 도출 및 `cnt` 누적 캐싱 도입.

- **Portfolio Selling Point:**
C++의 고질적인 메모리 초기화 결함을 이해하고 있으며, 다중 테스트 케이스 채점 환경을 대비한 전역 포인터 초기화 로직을 구축할 수 있음. 나아가 단순한 자료구조 암기를 넘어, 요구사항에 맞춰 노드 내부에 `cnt` 변수를 추가하여 검색 시간 복잡도를 혁신적으로 감축하는 설계 능력을 증명함.

### PRG Custom - 기지 건설 순서 정렬
- **Topic:** Graph | Topological Sort | Queue
- **Folder:** `Week_10/05_TopologicalSort/`
- **Time Complexity:** O(N + M) ( 모든 건물 N번 큐 삽입/추출, 모든 선행 규칙 M번 스캔. 32,000 + 100,000 = 132,000번 연산으로 1초 ㅔ한을 0.001초 만에 돌파. )
- **Space Complexity:** O(N + M) ( 건물 32,000개의 상태를 담는 배열과 100,000개의 간선을 담는 인접 리스트 벡터. 약 수 MB 이내로 극도로 가벼움. )

- **Key Idea:**
'선후 관계'가 명확한 데ㅣ터를 정렬할 때는 크기 비교 정렬( `std::sort` )이 아닌 방향 비순환 그래프( DAG ) 기반의 위상 정렬을 사용한다.
자신을 향해 들어오는 화살표의 개수를 의미하는 `indegree`( 진입 차수 ) 배열을 만든다. `indegree`가 0인 노드( 선행 조건이 없는 노드 )를 큐에 넣고 빼면서, 그 노드와 연결된 다음 노드들의 `indegree`를 1씩 깎는다( `indegree[next]--` ). 이 과정에서 새롭게 진입차수가 0이 된 노드를 다시 큐에 넣는 과정을 반복하면, 큐에서 빠져나온 순서가 곧 완벽한 선행 규칙 시퀀스가 된다.

- **Caution:**
건물의 번호는 1번부터 시작하므로 0-Based Indexing을 사용하면 배열 범위 초과나 마지막 노드 누락이 발생한다. 따라서 배열 크기를 `N + 1`로 패딩하고 루프를 `1 <= n`으로 제어하는 것이 필수적이다. 만약 큐가 모두 비워졌는데 반환된 결과 배열의 크기가 N과 다르다면, 어딘가에 교착 상태가 존재한다는 뜻이므로 실무에서는 예외를 던져야 한다.

- **Improvement:**
지문에서 "답이 여러 개일 경우 번호가 작은 건물부터 지으라"는 추가 조건이 붙는다면, 즉시 `queue<int> q;`를 최소 힙 기반의 `priority_queue<int ,vector<int>, greater<int>> pq`로 교체하여 아키텍처를 업그레이드해야 한다.

- **Problem Hint:**
1. 스케일 확인 : N = 32,000. M = 100,000. O(N^2) 금지.
2. 요구사항 분석 : "먼저 건설되어야 하는", "위배하지 않으면서 순서대로" -> 방향성 및 의존성 존재.
3. 알고리즘 도출 : DAG 환경에서의 시퀀스 정렬을 위한 `Topological Sort( 위상 정렬 )` 도출.

- **Portfolio Selling Point:**
단순히 수치 비교 정렬의 한계를 명확히 인지하고 있으며, 복잡한 시스템의 의존성 트리를 O(V + E)의 비용만으로 해결하는 위상 정렬 아키텍처를 자유자재로 구사할 수 있음. 특히 C++ 메모리 해제 시점을 정확히 이해하여 상태 동기화 버그를 사전 차단하는 백엔드/클라이언트 코어 프로그래머의 역량을 증명함.

### PRG Custom - 오픈월드 내비게이션 엔진
- **Topic:** Graph | Shortest Path | Dijkstra | Priority Queue
- **Folder:** `Week_11/01_Dijkstra/`
- **Time Complexity:** O(E log V) ( 모든 간선 E개를 스캔하며, 우선순위 큐에 넣고 빼는 비용이 log V만큼 소요. 300,000 * log2(20,000) ≒ 4,300,000번 연산으로 1초 제한 통과 )
- **Space Complexity:** O(V + E) ( 마을의 개수 20,000개의 상태를 담는 `dist` 배열과 간선 300,000개를 담는 `adj` 인접 리스트. 메모리 사용량 최적화 완료 )

- **Key Idea:**
특정 시작점에서 다른 모든 곳으로 가는 최단 거리를 구할 때는 BFS가 아닌 우선순위 큐 기반의 다익스트라를 사용한다. 가중치가 작은 경로를 항상 먼저 팝하기 위해 큐의 자료형은 최소 힙( `greater` )으로 설정한다. 핵심은 가지치기( Pruning )다. 큐에서 꺼낸 경로의 비용이 이미 내가 알고 있는 최단 거리 장부( `dist` )의 비용보다 높다면, 그 경로는 탐색할 가치가 없으므로 `continue`로 즉시 버린다. 이를 통해 폭발적인 중복 탐색을 막아낸다.

- **Caution:**
1. Dangling Reference : 큐의 최상단 요소를 변수에 담을 때 무심코 `const auto&`를 쓰면, 바로 다음 줄에서 `pq.pop()`을 하는 순간 참조하던 메모리가 날아가 프로그램이 터진다. 반드시 `const auto cur`로 값을 복사해 와야 한다.
2. 간선 적재 순서 : `priority_queue`는 `pair`의 첫 번째 원소를 기준으로 정렬한다. 따라서 인접 리스트 배열이나 큐에 데이터를 넣을 때 반드시 `{ 가중치, 목적지_노드 }` 순서로 `emplace` 해야 가중치 기반 정렬이 정상 작동한다.

- **Improvement:**
현재는 단순히 최소 '비용'만 구하지만, 실제 내비게이션 엔진에서는 '어떤 경로를 거쳐 갔는지'를 화면에 점선으로 그려주어야 한다. 이를 위해 1차원 배열 `vector<int> parent(v + 1);`를 추가하여, 거리가 갱신될 때마다 `parent[next.second] = cur.second;`를 저장해 두면, 도착지에서부터 역추적하여 최단 경로의 노드 히스토리를 완벽하게 복원할 수 있다.

- **Problem Hint:**
1. 스케일 확인 : V = 20,000. E = 300,000. O(V^2)를 요구하는 단순 2차원 배열 탐색은 즉각 시간 초과.
2. 요구사항 분석 : '시작 마을에서'( 단일 출발점 ). '모든 다른 마을들로 가는'( 다중 목적지 ). '최소 이동 비용'( 가중치 양수 ).
3. 알고리즘 도출 : 단일 출발 최단 경로 탐색에 특화된 `Dijkstra( 다익스트라 )` 알고리즘 및 O( E log V ) 최적화 도출.

- **Portfolio Selling Point:**
C++ STL의 얕은 복사와 깊은 복사 개념을 완벽하게 이해하고 있어 실무 메모리 관리에서 발생하는 크래시를 방지할 수 있으며, 그래프 릴렉세이션 매커니즘을 3분 이내에 코드로 직조해 낼 수 있는 최적화 기반 서버 아키텍트 역량 증명.

### PRG Custom - 물류망 최적화 엔진
- **Topic:** Graph | Shortest Path | Floyd-Warshall | Matrix
- **Folder:** `Week_11/02_FloydWarshall/`
- **Time Complexity:** O(N^3) ( 경유지 K, 출발지 I, 도착지 J가 이루는 3중 루프를 통해 100^3 = 1,000,000번의 고정 연산 수행. 현대 CPU에서 0.002초 이내 연산 종료 )
- **Space Complexity:** O(N^2) ( 101 * 101 크기의 2차원 고정 벡터 공간 사용. 공간 복잡도 최소화 달성 )

- **Key Idea:**
그래프의 모든 정점에서 다른 모든 정점으로의 최단 거리를 한 번에 구할 때는 플로이드-워셜 알고리즘을 사 용한다. 동적 계획법( DP )의 원리를 적용하여, `dist[i][j]` ( i에서 j로 바로 가는 거리 )를 `dist[i][k] + dist[k][j]` ( i에서 임의의 경유지 k를 거쳐 j로 가는 거리 )와 비교해 더 짧은 값으로 계속해서 장부를 갱신( `min` )해 나간다. 반드시 경유지 `k` 루프가 가장 바깥쪽에 윛치해야 모든 정점이 골고루 경유지로 검증될 수 있다.

- **Caution:**
입력 데이터에 동일한 출발지와 도착지를 가지면서 비용만 다른 중복 노선이 존재할 수 있으므로, 초기화 단계에서 `dist[e[0]][e[1]] = min(dist[e[0]][e[1]], e[2]);` 처리를 누락하면 낡은 데이터가 덮어씌워져 오답이 된다.

- **Improvement:**
만약 특정 경로나 경유 노선이 실제로 단 하나도 존재하지 않는 단절된 컴포넌트 구조가 많다면, 3중 루프 내부에 `if ( dist[i][j] == INF ) continue;` 구문을 삽입하여 J 루프 전체를 스킵하는 분기 예측 최적화를 적용하면 실전 연산 속도를 대폭 줄일 수 있다.

- **Problem Hint:**
1. 스케일 확인 : 노드의 개수 N <= 100 이하로 극단적으로 작음 -> O(N^3) 복잡도가 최적의 해법임을 강력하게 암시.
2. 요구사항 분석 : 단일 타겟이 아닌 "모든 센터에서 다른 모든 센터로의 최소 비용 매트릭스" 요구.
3. 알고리즘 도출 : 전 정점 최단 경로 도출의 표준 아키텍처임 `Floyd-Warshall` 도출.

- **Portfolio Selling Point:**
네트워크 전수 스캔 및 비용 테이블 작성을 다익스트라 N번 반복이라는 비효율적인 방식 대신, CPU 캐시 메모리 지역성을 극대화하는 정형 3중 루프 점화식으로 제어하여 클라이언트/서버 데이터베이스 통신 비용을 혁신적으로 압축하는 구현력 증명.

### PRG Custom - 초고속 통신망 구축 엔진
- **Topic:** Graph | Minimum Spanning Tree ( MST ) | Kruskal | Union-Find
- **Folder:** `Week_10/06_UnionFind_Kruskal/`
- **Time Complexity:** O(M log M) ( 가장 무거운 연산은 간선을 정렬하는 데 쓰이는 M log M임. 이후 Union-Find는 O(1)에 수렴하므로, 최대 10만 개의 간선이 주어져도 약 170만 번 연산으로 2초 제한을 가볍게 돌파 )
- **Space Complexity:** O(N) ( 기지의 개수 N만큼의 상태를 담는 `parent` 배열 사용 )

- **Key Idea:**
"모든 정점을 연결하는 최소 비용"이라는 키워드는 최소 신장 트리( MST )를 요구한다. 이를 구현하는 가장 직고나적인 방법인 크루스칼 알고리즘은 그리디 접근법을 쓴다. 비용이 싼 간선부터 무조건 차례대로 연결하되, '사이클이 생기면 안 된다'는 절대 규칙을 지켜야 한다. 이를 위해 두 노드의 최상위 부모를 확인하는 `Find` 연산과, 부모가 다를 때만 합치는 `Union` 연산을 결합하여 사이클 발생 여부를 실시간으로 판별한다.

- **Caution:**
1. `Find` 함수에서 단순히 부모를 찾는 것에 그치지 않고 `return parent[x] = Find(parent[x]);`로 경로 압축을 하지 않으면, 나중에 노드가 일렬로 길게 늘어질 때 탐색 시간이 초과된다.
2. `Union` 함수 내부에서 직속 부모끼리 비교( `a = parent[a]` )하면 안 되고, 반드시 최상위 루트( `a = Find(a)` )를 끌어와서 비교해야 집합 병합 논리가 성립한다.

- **Improvement:**
현재 코드는 부모를 병합할 때 무조건 `b`의 부모를 `a`로 만든고 있다. 실무 엔진에서는 트리의 깊이가 깊어지는 것을 완벽하게 방어하기 위해, 트리의 높이를 기록하는 배열을 하나 더 두어 '항상 높이가 낮은 트리를 높이가 높은 트리 밑으로 붙이는' Rank By Union 기법을 추가할 수 있다. ( 단, 코딩 테스트 스케일에서는 경로 압축만으로도 충분히 빠르다. )

- **Problem Hint:**
1. 스케일 확인 : N = 1,000, M = 100,000. O(M log M) 정렬 후 탐색 가능.
2. 요구사항 분석 : "모든 기지가 통신 가능하도록", "전체 선로 건설 비용 최소화" -> 전형적인 MST 요구사항.
3. 알고리즘 도출 : 최소 비용 간선부터 탐색하는 `Kruskal` 알고리즘 및 사이클 판별용  `Union-Find` 구조체 도출.

- **Portfolio Selling Point:**
복잡한 그래프 연결 상태를 `int parent 배열` 하나만으로 통제하는 효율적인 메모리 설계 능력을 입증함. 특히 수학적 명제( MST 간선의 수는 N-1개 )를 활용해 불필요한 루프를 조기에 끊어내는 `Early Exit` 최적화 감각은 대규모 트래픽을 다루는 게임 서버 프로그래머에게 요구되는 핵심 역량임.

- **Topic:** Binary Search | Parametric Search | Optimization to Decision
- **Folder:** `Week_11/03_ParametricSearch/`
- **Time Complexity:** O(N log(max(T) * M)) ( 최대 탐색 공간이 10^18이므로 이분 탐색 루프는 최대 약 60회 가동됨. 그 안에서 N( 100,000 ) 순회가 일어나므로 총 연산량 약 6,000,000번으로 1초 제한을 여유롭게 통과 )
- **Space Complexity:** O(1) ( 주어진 배열 외에 추가적인 메모리 할당이 없어 공간 복잡도 제로에 수렴 )

- **Key Idea:**
탐색의 대상이 '배열의 인덱스'가 아니라 '정답( 시간 ) 자체'가 되는 패러매트릭 서치의 정수. "M명을 처리하는 최소 시간을 구하라"는 최적화 문제를, "X초가 주어질 때 M명을 처리할 수 있는가?"라는 결정 문제로 전환한다. 정답이 가능한 시간이라면 탐색 범위를 왼쪽( 더 짧은 시간 )으로 좁히고( `high = mid - 1` ), 불가능하다면 오른쪽( 더 긴 시간 )으로 넓히며( `low = mid + 1` ) 최적의 해를 찾아 들어간다.

- **Caution:**
1. 자료형의 한계 : `high`의 최댓값이 10^18에 달하므로, 시간과 관련된 모든 변수( `low`, `high`, `mid`, `answer`, `cnt` )는 반드시 `int`가 아닌 `long long`으로 선언해야 한다.
2. 이분 탐색 무한 루프 : `while( low <= high )`와 `answer` 변수 저장 구조를 사용하는 것이 실전에서 가장 안전하다.

- **Improvement:**
`isPossible` 함수의 인자로 `vector<int>& times`를 넘겼으나, 실무 C++ 엔진에서는 읽기 전용 참조임을 명확히 하고 우발적인 값 변경을 막기 위해 `const vector<int>& times`로 상수화하여 넘기는 습관을 들이는 것이 더 완벽하다.

- **Problem Hint:**
1. 스케일 확인 : 유저 수 M = 1,000,000,000. 반복문이나 큐를 이용한 1:1 시뮬레이션 절대 불가
2. 요구사항 분석 : "최소 시간을 구하라" & 특정 시간을 기준으로 성공/실패가 명확히 갈림( 단조성 ).
3. 알고리즘 도출 : 극단적 스케일의 탐색 공간을 O(log X)로 압축하는 `Parametric Search` 도출.

- **Portfolio Selling Point:**
알고리즘의 형태를 최적화 문제에서 결정 문제로 유연하게 뒤집을 수 있는 사고방식을 증명함. 나아가 C++ 엔진의 치명적 결함인 정수 오버플로우를 사전에 인지하여 `long long` 자료형의 맵핑과 조기 종료 최적화를 능숙하게 구현해 냄.

**PRG Custom - 거래소 매물 스캐너**

### PRG Custom - 거래소 매물 스캐너
- **Topic:** Two Pointers | Sliding Window | Prefix Sum Optimization
- **Folder:** `Week_11/04_TwoPointers/`
- **Time Complexity:** O(N) ( `left`와 `right` 포인터가 배열을 뒤로 후진하는 일 없이 오직 전진만 하므로, 각 원소는 최대 한 번 더해지고 한 번 빼짐. N = 100,000일 때 약 20만 번의 연산으로 1초 제한을 여유롭게 관통 )
- **Space Complexity:** O(1) ( 별도의 자료구조 없이 포인터 인덱스 변수 2개만 사용 )

- **Key Idea:**
배열의 원소가 모두 양수라는 절대 규칙이 존재할 때만 사용 가능한 최상위 스캐닝 기법. `right` 포인터를 밀면 합이 무조건 '증가'하고, `left` 포인터를 당기면 합이 무조건 '감소'한다는 단조성을 십분 활용한다.
`sum < k` 라면 먹이가 부족하므로 우측으로 뱀의 머리( `right` )를 뻗어 집어삼키고, `sum > k` 라면 과식했으므로 뱀의 꼬리( `left` )를 잘라내어 소화시킨다. 이 뱀이 배열을 단 한 번 스치고 지나가는 것 만으로 모든 구간 조합을 O(N)에 평가할 수 있다.

- **Caution:**
1. Out of Bounds 방어 : `right`가 배열의 끝( `n - 1` )에 도달했는데 `sum < k`인 경우, 더 이상 더할 매물이 없음에도 `sequence[++right]`를 호출하면 쓰레기 값을 참조하여 엔진이 크래시 난다. 반드시 `if ( right == n - 1 ) break;`로 멈춰 세워야 한다.
2. 갱신 누락 : 길이가 더 짧은 구간을 찾았을 때, 반드시 `min_len` 장부를 갱신해야 뒤따라오는 무가치한 긴 구간의 덮어씌우기를 방어할 수 있다.

- **Improvement:**
현재 코드는 포인터 역전을 막기 위해 `left <= right` 조건을 `while`문에 걸어두었다. 하지만 매물이 0보다 큰 양수라면, `sum < k` 일때는 무조건 `right`가 전진하고, `sum > k` 일 때는 무조건 `left`가 쫓아오며, `left`가 `right`를 추월하는 순간 `sum = 0`이 되어 다시 `right`가 도망가게 되는 자가 치유 논리가 성립한다. 따라서 특별한 예외가 없다면 `while ( right < n )`만으로도 엔진은 안전하게 돌아간다.

- **Problem Hint:**
1. 스케일 확인 : N <= 100,000. 이중 `for`문 O(N^2) 사용 불가. 선형 탐색 요구.
2. 요구사항 분석 : "연속된 구간의 합", "원소는 양수", "가장 짧은 길이", "가장 앞쪽 인덱스".
3. 알고리즘 도출 : 양수 배열의 구간 조작 단조성을 활용하는 `Two Pointers` 도출.

- **Portfolio Selling Point:** 
이중 루프의 브루트 포스 탐색이 일으키는 병목 현상을 정확히 인지하고, 실시간으로 늘어났다 줄어드는 동적 윈도우를 설계하여 연산량을 기하급수적으로 압축하는 코어 클라이언트/서버 프로그래머의 자질을 입증함.
